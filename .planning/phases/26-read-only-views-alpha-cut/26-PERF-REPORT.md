# Phase 26 Perf Report — NF-01 (60fps Skills view, 2000-skill library)

## Target

- **p95 inter-frame delta < 18ms** (~55fps p95) sustained for 2 seconds of search-as-you-type
- **Fixture:** 2000 synthetic skills generated by `crates/tome-desktop/tests/perf/synthetic_skills.rs` (random 100–5000 char SKILL.md bodies, deterministic seeded RNG)
- **Hardware:** macos-latest GitHub Actions runner (Apple Silicon since the 2024 migration — Assumption A12 in 26-RESEARCH.md)
- **Browser:** Playwright-bundled Chromium (fallback: system Chrome via `PW_USE_SYSTEM_CHROME=1`)
- **Virtualizer:** React Aria native `<Virtualizer layout={ListLayout} layoutOptions={{ rowSize: 52, ... }}>` — OQ-1 path A, locked in plan 26-02

## OQ-1 status

**Path A confirmed (so far).** React Aria's native `<Virtualizer>` wins on:

1. Zero extra dependency (already pulled in for the rest of the React Aria primitives).
2. Free a11y semantics — `role="listbox"` + `role="option"` for screen-reader navigation come from the same primitives that drive the Sidebar.
3. Simpler API for fixed 52px rows — no measurement complexity.

**Fallback (path B):** if a later run fails p95 < 18ms and the regression doesn't point at a different fix, swap the `<Virtualizer>` + `<ListBox>` in `SkillsView.tsx` for `@tanstack/react-virtual` per 26-RESEARCH.md §"Standard Stack — Virtualisation Path B". The CSS-Modules row styles + the row component itself (`<SkillListRow>`) remain reusable; only the outer scrolling-window wrapper changes.

## Methodology

1. Fixture generator writes `${PERF_FIXTURE_OUT}/perf-skills.json` — a flat array of 2000 `DiscoveredSkill`-shaped objects.
2. Bench harness copies that file to `crates/tome-desktop/ui/src/__mocks__/perf-skills.json` so the Vite-aliased Tauri mock returns it from `commands.listSkills`.
3. Vite builds the UI bundle with `PERF_TEST=1` → `import.meta.env.VITE_PERF_TEST="1"` in the mock → the 2000-skill fixture is loaded instead of the 3-skill a11y fixture.
4. Playwright spec navigates to the Skills view, waits for the `<ListBox>` to render, focuses the SearchField, calls `window.__startFpsSampler(2000)` to reset the frame buffer, types `"tdd"` at 15ms inter-keystroke delay, waits 2.5s, reads `window.__fpsFrames`, computes p50/p95/max.
5. p95 < 18ms is the pass criterion.

## OQ-1 evaluation — first-run boundary case

**Local-hardware baseline (Apple Silicon M-series Mac, Playwright-bundled Chromium, headed dev mode):** the four runs above cluster at:

- **p50 = 16.70ms** — exactly vsync 60Hz. The steady-state per-frame work is well within budget.
- **p95 = 18.10–18.40ms** — consistently 0.1–0.4ms above the 18ms target. Translates to ~54–55fps p95, vs. the 55fps p95 the threshold encodes.
- **max = 18.60–31.73ms** — one or two ~30ms stalls per 2s window (likely React reconciliation cost on the keystroke that materialises the first-page virtualised rows), then steady ~18ms second-vsync drops.
- **Dropped frames (>18ms) = 7–13 of 121 samples (5.8–10.7%).** The 95th-percentile-of-deltas metric lands at the edge.

**Interpretation:** This is **NOT a clear failure** of the React Aria native `<Virtualizer>` — the virtualisation is working (p50 is locked at 60fps). The boundary is in the **tail behaviour during search-as-you-type**: the keystroke handler + fuse.js filter pass + re-render cycle costs one or two ~one-vsync stalls per keystroke. The TanStack-Virtual swap (path B) would change the renderer's scrolling-window math but would NOT eliminate the keystroke-driven filter-and-render cost — that's a separate optimisation surface (debouncing the filter, memoising the row component harder, or splitting the work into a transition).

**Two interpretations of the boundary:**

1. **"The threshold is tuned a hair too tight for the React Aria path."** The plan's own commentary calls 60fps over 1s unrealistic and chose 18ms as a slack-bearing approximation. The empirical local-hardware result puts us 0.1–0.4ms past that approximation — well inside measurement noise on a contended laptop with Vite's dev server, hot-module-reload watcher, and browser dev-tools running. A 19ms (~52fps p95) threshold would land all four runs as PASS. Plan-2-fwd: re-baseline once CI gives clean numbers on macos-latest dedicated Apple Silicon (Assumption A12).

2. **"The React Aria path has a measurable steady-state regression vs path B."** TanStack Virtual is the documented fallback (26-RESEARCH.md §"Standard Stack — Virtualisation"). It's a non-trivial SkillsView refactor — out of scope for plan 26-08 per executor instructions, but ON THE TABLE if CI numbers also show the boundary failure.

**Recommended next step:** let the CI workflow run on macos-latest (Task 3) carry the source-of-truth measurement. Dedicated-hardware numbers should be either (a) cleanly under 18ms — confirming the local-hardware result was contention noise — or (b) showing the same boundary pattern — at which point the TanStack swap becomes the called-for fix and gets filed as the first phase-27 issue.

**No silent threshold change.** The 18ms target in `60fps-search.spec.ts` is unchanged. CI will fail on PR if the result lands above it — that failure surface is the signal the plan asked for.

## Implementation choices captured in the harness

- **`addInitScript` loads `fps-sampler.js`** (plain JS, not TS) because Playwright reads the file verbatim into the page — no transpilation. TypeScript globals for `window.__fpsFrames` + `window.__startFpsSampler` are declared at the top of `60fps-search.spec.ts` for type-checked `page.evaluate` calls.
- **Manual sampler start.** The sampler doesn't auto-run from page load — including the first-paint + initial-list-render cost would distort the median. The spec calls `__startFpsSampler(2000)` immediately before `keyboard.type()` so the window covers steady-state typing only.
- **Vite alias gated by `PERF_TEST=1`.** Mirrors plan 26-07's `A11Y_TEST=1` pattern; both modes share the same mock modules. `commands.listSkills` branches on `import.meta.env.VITE_PERF_TEST === "1"` (pre-substituted at build time via Vite's `define`) to switch between the 3-skill a11y fixture and the 2000-skill perf fixture.
- **In-tree stub `perf-skills.json`** (3 placeholder skills) ships in `crates/tome-desktop/ui/src/__mocks__/` so unrelated UI builds + the a11y gate keep resolving the import. The bench harness copies the real `${PERF_FIXTURE_OUT}/perf-skills.json` over the stub before `npm run dev:perf`.
- **Runs log written to `26-PERF-RUNS.tsv`**, not appended to this report. Naive markdown append to the report's middle table lands rows after later prose sections — confusing. The TSV is gitignored; the GitHub Actions workflow uploads it as an artefact, and curated rows are promoted into the "Baseline runs" table above during human review.
