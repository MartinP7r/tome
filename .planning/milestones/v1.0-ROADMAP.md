## v1.0 Phase Map

Seven phases. Phases 10–11 produce the alpha (read-only desktop client). Phases 12–13 produce the beta (sync + config UI = daily-driver). Phases 14–15 produce the rc (all mutating commands). Phase 16 closes distribution and ships v1.0.

## Phase Details

### Phase 10: Rust Core Extraction + Tauri Integration Spike

**Goal:** Reshape `crates/tome` so its domain operations return structured types callable from any front-end, add `crates/tome-desktop` as a sibling Tauri app crate, and prove the architecture with a working "hello tome" window that calls into the real `status::collect`.

**Requirements:** CORE-01, CORE-02, CORE-03, CORE-04, CORE-05

**Depends on:** v0.9 shipped (or v0.9 swap with v1.0 — see D-GUI-09).

**Why first:** Every other GUI requirement depends on the Rust↔JS contract. The CLI's `lib.rs::run` currently mixes domain logic with stdout formatting; until that's untangled, the GUI either duplicates logic or scrapes formatted strings. This phase pays the architectural debt once, for both CLI and GUI.

**Success Criteria** (what must be TRUE):
1. Each top-level CLI command has a corresponding domain function returning a structured Rust type (`StatusReport`, `ListReport`, `LintReport`, `RemovePlan`, etc.). Existing CLI is rewired to call these and format their output; integration tests still pass.
2. `crates/tome-desktop` builds a debug `.app` on macOS via `cargo tauri dev`. The app opens a window that displays a real `StatusReport` from `tome::status::collect` against the user's actual `tome_home`.
3. `bindings.ts` is generated by `specta` + `tauri-specta` and contains TypeScript definitions for every type that crosses the IPC boundary. CI fails if the generated bindings are out of date.
4. Long-running operations emit at least one progress event per stage; the spike app subscribes to `sync-progress` events and renders a placeholder progress bar.
5. Every domain function returns `Result<T, TomeError>` where `TomeError` carries a stable enum `code` and a human-readable `message`. Front-end can pattern-match on `code` without inspecting the message string.

**Frontend framework decision:** Resolves D-GUI-04. Spike builds the same status-list view in two of the three candidates (React, Solid, Svelte) and the team picks one based on dev-loop ergonomics, build size, and bindings ergonomics.

**Plans (TBD at `/gsd:plan-phase`):**
- 10-01: Decompose `lib.rs::run` into CLI presenter + domain calls (CORE-01)
- 10-02: Add `crates/tome-desktop` workspace member with Tauri 2 scaffolding (CORE-02)
- 10-03: Wire `specta` + `tauri-specta` for `bindings.ts` generation (CORE-03)
- 10-04: Tauri event channel for progress reporting (CORE-04)
- 10-05: `TomeError` enum with stable codes; threading through `Result` chain (CORE-05)
- 10-06: Frontend framework spike + decision

**UI hint:** spike-quality only. No production UI in this phase.

### Phase 11: Read-Only Views (Alpha cut)

**Goal:** Ship the read-only half of the GUI: status dashboard, virtualised skill list, detail pane with markdown preview, doctor health view, and on-disk file watcher. After this phase, the desktop app is useful for inspection but does not mutate state.

**Requirements:** VIEW-01, VIEW-02, VIEW-03, VIEW-04, VIEW-05, VIEW-06 + NF-01 (perf budget) + NF-02 (a11y) + NF-03 (HIG) + NF-05 (concurrency).

**Depends on:** Phase 10.

**Success Criteria:**
1. Status dashboard renders all fields from `tome status --json` with role/type badges, last-sync time, and lockfile state. Refreshes within 200 ms of an external `tome sync` from the CLI.
2. Skill list view handles a 2000-skill library at sustained 60 fps during search-as-you-type on M1 (8 GB) — verified via a synthetic-skills bench in CI.
3. Detail pane renders frontmatter (parsed by existing `lint.rs`), source path, content hash, last sync, managed/local badge, and disabled state. Three actions (open source, copy path, disable on this machine) wired to the same handlers as the existing browse TUI.
4. Markdown preview renders SKILL.md body with the same Markdown subset as `browse/markdown.rs`. Code blocks, headings, lists, links all render.
5. Health pane lists all `tome doctor` findings with one-click fix actions. Fix actions go through the same repair handlers used by interactive `tome doctor`.
6. File watcher reloads UI state when manifest, lockfile, or library content changes externally. No stale UI after CLI sync.
7. Every interactive element is keyboard-accessible and has a VoiceOver label. Native menu bar shows File / Edit / View / Library / Help.

**Cut:** **v1.0-alpha**. Internal release; CLI still required for sync and edit.

**Plans (TBD):**
- 11-01: Status dashboard view (VIEW-01)
- 11-02: Virtualised skill list + fuzzy search (VIEW-02)
- 11-03: Detail pane + per-skill actions (VIEW-03)
- 11-04: Markdown preview component (VIEW-04)
- 11-05: Doctor health pane + fix actions (VIEW-05)
- 11-06: File watcher integration + auto-refresh (VIEW-06)
- 11-07: A11y + HIG audit (NF-02, NF-03)
- 11-08: Perf bench + harness (NF-01)

**UI hint:** yes — first user-visible UI.

### Phase 12: Sync + Triage UI

**Goal:** Replace `tome sync` and its interactive triage flow with a visual flow showing per-stage progress, lockfile diff, per-skill triage decisions, and previewable `machine.toml` writes. Highest-UX-risk phase.

**Requirements:** SYNC-01, SYNC-02, SYNC-03, SYNC-04, SYNC-05.

**Depends on:** Phase 11 (uses the file watcher and detail pane).

**Success Criteria:**
1. "Sync" action runs the full pipeline (discover → consolidate → distribute → cleanup → save) with per-stage progress and current-directory indicator. CLI's interactive triage is *not* invoked — the GUI's triage panel replaces it.
2. Lockfile diff produces a triage panel listing new / changed / removed skills. Per-skill default action is "keep"; alternates are "disable on this machine" and (for git skills) "view source". Bulk actions (e.g. "disable all new from `<directory>`") work.
3. Triage decisions render as a `machine.toml` diff before save. User clicks "apply" to write; "cancel" abandons without side-effects.
4. Cancel action during sync stops the pipeline at the current stage boundary and leaves library state consistent (no half-written manifest, no partial lockfile). Verified via integration test.
5. Failed sync surfaces a per-stage failure summary with retry. Retry resumes from the failed stage where possible (re-running discover + consolidate is acceptable; rerunning distribute on a partial manifest is not).

**Plans (TBD):**
- 12-01: Per-stage progress channel + UI (SYNC-01)
- 12-02: Triage panel with lockfile diff (SYNC-02)
- 12-03: `machine.toml` diff preview (SYNC-03)
- 12-04: Cancellation + consistency tests (SYNC-04)
- 12-05: Failure summary + retry (SYNC-05)

**UI hint:** yes — most novel UI surface in v1.0.

### Phase 13: Configuration UI (Beta cut)

**Goal:** Replace `tome init` + `tome add` + hand-edited `tome.toml`/`machine.toml` with a GUI-driven configuration experience. After this phase, the user can set up tome from scratch and add/edit/remove directories without touching a terminal.

**Requirements:** CFG-01, CFG-02, CFG-03, CFG-04, CFG-05 + NF-04 (destructive confirms).

**Depends on:** Phase 12 (sync UI required to validate config changes end-to-end).

**Success Criteria:**
1. First-run experience launches when no `tome.toml` exists at the resolved `tome_home`. Greenfield/brownfield/legacy semantics from WUX-01..05 apply (the GUI's first-run gates the same machine-states as the CLI wizard).
2. Directory editor lists all configured directories; user can add (auto-discovered + custom), edit, remove (with cleanup preview), and reorder. Type/role combos validate live against `valid_roles()`. Path overlap (Cases A/B/C from `Config::validate()`) errors render inline before save.
3. "Add git repository" form captures URL + optional name + ref pinning. Clone progress streams via the SYNC-01 event channel. Failure surfaces inline with retry.
4. Machine preferences editor exposes `disabled` skills, `disabled_directories`, and per-directory `enabled`/`disabled` lists. Changes preview as a `machine.toml` diff before save.
5. Every config save routes through `Config::save_checked`. The GUI cannot produce a config the CLI rejects. Verified via integration test that fuzzes invalid combos and asserts they're blocked.

**Cut:** **v1.0-beta**. Daily-driver capable for the primary user.

**Plans (TBD):**
- 13-01: First-run experience + greenfield/brownfield/legacy gates (CFG-01)
- 13-02: Directory editor with live validation (CFG-02)
- 13-03: Add-git-repo form + clone progress (CFG-03)
- 13-04: Machine prefs editor + diff preview (CFG-04)
- 13-05: `Config::save_checked` enforcement + invalid-combo fuzz tests (CFG-05)

**UI hint:** yes.

### Phase 14: Mutating Operations UI

**Goal:** Bring the remaining destructive CLI commands (`tome remove`, `tome reassign`, `tome fork`, `tome relocate`, `tome eject`) into the GUI with plan/preview/confirm flows.

**Requirements:** OPS-01, OPS-02, OPS-03, OPS-04 + NF-04 (destructive confirms).

**Depends on:** Phase 13.

**Success Criteria:**
1. Removing a directory shows a `RemovePlan` preview itemising affected symlinks, library dirs, manifest entries, and lockfile entries. Confirmation is required before execute. Partial-failure aggregation (SAFE-01 semantics) renders with retry-per-item.
2. Reassign / fork flows let the user pick destination from a dropdown of compatible directories. Plan preview shows what changes; execute is confirmed.
3. Relocate flow previews target path and symlink-rewrite count; runs atomically; failure leaves library at original path.
4. Eject shows count of distribution symlinks to remove with a confirmation; reversible by re-running sync.

**Cut:** **v1.0-rc**. CLI is functionally optional for steady-state use.

**Plans (TBD):**
- 14-01: Remove flow with `RemovePlan` preview + partial-failure UI (OPS-01)
- 14-02: Reassign + fork flows (OPS-02)
- 14-03: Relocate flow (OPS-03)
- 14-04: Eject flow (OPS-04)

**UI hint:** yes.

### Phase 15: Backup UI

**Goal:** Replace the five `tome backup` subcommands with a coherent history-and-restore UI.

**Requirements:** BAK-01, BAK-02, BAK-03, BAK-04 + NF-04 (destructive confirms on restore).

**Depends on:** Phase 14.

**Success Criteria:**
1. Backup history view shows commits in the library: timestamp, message, SHA. Pagination matches `tome backup list -n` semantics.
2. Snapshot action prompts for message (default: timestamp), runs `git add . && git commit`, and surfaces success/failure as a toast.
3. Diff view (against a selected commit, default `HEAD`) shows changed skills with per-file diff rendering.
4. Restore flow previews affected skills and lockfile state, requires confirmation, and triggers an automatic sync afterward to reconcile distribution targets. Verified via integration test that confirms the post-restore sync runs.

**Plans (TBD):**
- 15-01: Backup history view (BAK-01)
- 15-02: Snapshot action + toast (BAK-02)
- 15-03: Diff view (BAK-03)
- 15-04: Restore flow + post-restore sync (BAK-04)

**UI hint:** yes.

### Phase 16: Distribution (v1.0 ship)

**Goal:** Code sign, notarize, package, auto-update, and ship v1.0. CLI release flow is not regressed.

**Requirements:** DIST-01, DIST-02, DIST-03, DIST-04, DIST-05.

**Depends on:** Phase 15.

**Success Criteria:**
1. Release builds produce signed + notarized + stapled DMGs for `aarch64-apple-darwin` and `x86_64-apple-darwin` (or one universal binary per current Tauri/Apple guidance). Gatekeeper accepts the DMG without warnings.
2. Auto-update via `tauri-plugin-updater` against a signed manifest on GitHub Releases. User opt-in; updates surface as a non-blocking toast.
3. GitHub Actions release workflow ships GUI artifacts alongside CLI artifacts. CLI cargo-dist outputs unchanged.
4. First-launch UX explains tome's mental model in one screen, links to docs, and offers either CFG-01 wizard or "import existing `tome.toml`". Hardened runtime is enabled with the minimum entitlements; library validation stays on.
5. App embeds (or shells out to) the `tome` CLI binary; "Show in terminal" UX on every view copies a working command.

**Plans (TBD):**
- 16-01: macOS code-sign + notarize + DMG pipeline (DIST-01)
- 16-02: `tauri-plugin-updater` integration + signed manifest (DIST-02)
- 16-03: Combined release workflow (DIST-03)
- 16-04: First-launch UX + onboarding (DIST-04)
- 16-05: CLI bundling + "Show in terminal" affordances (DIST-05)
- 16-06: Pre-ship audit (NF-01..05)

**UI hint:** yes — onboarding screens are the last user-visible polish surface before ship.

## Open Questions

To resolve at `/gsd:new-milestone` ratification or in Phase 10:

- **Q1**: Frontend framework (D-GUI-04 still open). Recommend resolving in Phase 10 spike.
- **Q2**: Tauri minor version pinning policy — track `2.x` patch releases, or pin to a specific `2.M.N`?
- **Q3**: How does the GUI surface `tome lint` failures? As a separate view, integrated into doctor (VIEW-05), or both?
- **Q4**: Should the desktop app have a tray-icon presence (always-running)? Default position: **no**, launches on demand. Revisit at GUI-WATCH-01 (v2).
- **Q5**: Does v1.0 ship Linux as a stretch goal, or is Linux strictly v2? Default position: **strict v2** per D-GUI-06.
- **Q6**: Sparkle vs `tauri-plugin-updater` — Tauri's plugin is the default in D-GUI-05, but Sparkle is the macOS standard if Tauri's auto-update has rough edges in 2026. Verify in Phase 16.
- **Q7**: Telemetry / crash reporting — out of scope for v1.0, but flag for v2 (privacy-first, opt-in). Sentry-style or none?

## Sequencing alternatives (D-GUI-09)

Default: **v0.9 → v1.0**. v0.9 is small (one epic) and hardens `machine.toml` semantics that v1.0 will lean on heavily.

Alternative: **v1.0 → v0.9**. Acceptable if cross-machine portability becomes lower-priority than shipping a GUI. v1.0 would inherit the current `machine.toml` shape, and PORT-01..04 would land in v1.1.

Alternative not recommended: **v0.9 + v1.0 in parallel**. Single-developer project; context-switching cost outweighs parallelism gains.
