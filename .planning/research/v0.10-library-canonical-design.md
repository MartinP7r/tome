# v0.10 Design Exploration — Library-Canonical Model + Lockfile-Authoritative Sync

**Project:** tome v0.10 — Library-Canonical Model + Cross-Machine Plugin Reconciliation
**Domain:** Skill library state model, lockfile semantics, marketplace adapter pattern
**Researched:** 2026-05-01
**Status:** Draft — pending `/gsd:new-milestone` ratification
**Confidence:** MEDIUM-HIGH (design); HIGH (claude CLI surface, verified)

## Executive Summary

The v1.0 GUI milestone (currently drafted under `milestones/v1.0-*.md`) should be deferred. A more fundamental problem surfaced during the post-v0.9 codebase review: tome's library is a *consolidated cache*, but the project's primary user runs a **library-as-dotfiles** workflow where the library lives in a git-tracked `coding-agent-files/` repo and is expected to be portable across machines. Today's model violates this expectation in three ways:

1. **Managed skills** (Claude marketplace, git) are stored as **symlinks** into machine-specific cache paths (`~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/...`). Library committed to git → symlinks dangle on other machines.
2. **Source removal deletes library content** silently. `tome sync` cleanup phase removes any library entry whose source directory was removed from `tome.toml`. The user's `personal-skills` example (15 skills, real content, sourced from a now-removed directory) demonstrated this directly.
3. **Plugin uninstall or version churn breaks the library.** When Claude Code GCs old plugin versions, library symlinks point to deleted paths until next sync.

The proposed v0.10 reshapes the model: library becomes the **single source of truth**, all entries are real directory copies, the lockfile is **authoritative** for cross-machine reproducibility (Cargo.lock-shaped), and a **marketplace adapter trait** lets `tome sync` install/update plugins automatically when reconciling Machine B against the lockfile.

This is a meaningfully bigger milestone than the original v0.10 doc-drift cleanup, and it must land before the GUI: GUI types crossing the IPC boundary (`SkillEntry`, `LockEntry`, `RemovePlan`, etc.) change shape under the new model. Settling the contract first lets the GUI build on stable ground.

## Context

### Discussion lineage

This proposal arose organically from a single thread:

1. **PR #484 codebase review** surfaced 19 follow-up issues (#485–#503) and re-opened the question of milestone shape.
2. **"15 skill(s) from sources no longer configured"** — the user asked what this `tome sync` message meant. Diagnosis: the message lumps three distinct cases into one, the most common of which (source directory removed from config) deletes library content silently.
3. **"copy-audit is right there in the library, why is tome deleting it?"** — concrete reproduction of case #2. 15 skills in `~/dev/coding-agent-files/.claude/skills/` are marked stale because the user removed the `personal-skills` directory entry from `tome.toml` at some point.
4. **"shouldn't `~/dev/coding-agent-files/skills` be part of the library?"** — the design question. Should the library itself count as a source for skills it already contains?
5. **"managed skills should also be copies for resilience"** — the user's strengthening observation. Today's symlink model fails when plugins are uninstalled, version-bumped, or absent on a different machine.
6. **"on new machines, install/update needs to run on sync"** — the cross-machine reproducibility requirement. Lockfile becomes authoritative.

`~/dev/coding-agent-files/` is committed to git and stow-managed across machines. Today, pulling this repo on Machine B yields **62 dangling symlinks** in the library because Machine B's plugin cache layout differs.

### Verified `claude` CLI surface (open question #1 resolution)

Confirmed via `claude plugin --help`:

| Command | Programmatic? | Version pin? | JSON? | Notes |
|---|---|---|---|---|
| `claude plugin install <plugin>@<marketplace>` | ✓ | ✗ (latest only) | — | `-s, --scope user\|project\|local` |
| `claude plugin update <plugin>` | ✓ | ✗ (latest only) | — | "restart required to apply" |
| `claude plugin list --json` | ✓ | — | ✓ | returns `id`, `version`, `installPath`, `installedAt`, `enabled`, `scope` |
| `claude plugin uninstall <plugin>` | ✓ | — | — | |
| `claude plugin marketplace ...` | ✓ | — | — | manage available marketplaces |

**Hard constraint:** No `--version` flag on install/update. The adapter can ensure "something is installed" but cannot pin to a specific older version. Machine B will always go to *latest*, not lockfile-recorded version.

**Mitigation:** Lockfile records the version that was installed; adapter detects drift; sync surfaces "Machine B currently has 5.0.5, lockfile records 5.0.7, ran update → now at 5.0.8" as a warning. True version pinning is an upstream Claude Code feature request.

## Proposed Model

### Library is the single source of truth

```
sources →(update channel)→ library →(distribute)→ targets
                            ↑↑↑
                       ALL real directories
                       managed AND local both copies
                       portable across machines
                       survives source loss
```

- Both managed and local skills are **real directories** in `library_dir/`.
- Symlinks-only-in-distribution-targets (`~/.claude/skills/<name>` → `library_dir/<name>` is unchanged).
- `Manifest.managed: bool` becomes "update channel semantics" not "stored as symlink." Field name kept for compat; meaning shifted.

### Lockfile is authoritative for cross-machine reproducibility

`tome.lock` records per skill:

```json
{
  "source_name": "claude-plugins",
  "registry_id": "superpowers@claude-plugins-official",
  "version": "5.0.7",
  "git_commit_sha": "abc123...",
  "content_hash": "sha256:..."
}
```

`tome sync` flow becomes Cargo-shaped:

1. Load `tome.lock` (the truth).
2. For each managed entry: ask the marketplace adapter for current installed state.
3. **Match / Drift / Vanished** classification.
4. Render plan, confirm, resolve drift via `adapter.install` / `adapter.update`.
5. Re-discover, copy fresh content into library, recompute content_hash.
6. Distribute (unchanged).

### Marketplace adapter trait

```rust
pub trait MarketplaceAdapter {
    fn id(&self) -> &str;  // matches the path component: "claude-plugins-official", "axiom-marketplace"
    fn current_version(&self, plugin: &PluginRef) -> Result<Option<String>>;
    fn install(&self, plugin: &PluginRef) -> Result<InstallResult>;  // "install latest" — see constraint
    fn update(&self, plugin: &PluginRef) -> Result<InstallResult>;
    fn list_installed(&self) -> Result<Vec<InstalledPlugin>>;
    fn available(&self, plugin: &PluginRef) -> Result<bool>;
}
```

v0.10 ships:
- `ClaudeMarketplaceAdapter` — shells out to `claude plugin {install,update,list,uninstall}`. Parses `claude plugin list --json`. Uses the existing `installed_plugins.json` discovery as a fallback for plugin metadata.
- `GitAdapter` — wraps `crates/tome/src/git.rs::clone_repo` / `update_repo` (already exists) into the trait shape.

Future adapters (deferred, post-v0.10): `npm`, generic-URL, registries with proper version pinning.

### Source removal preserves library content

The "no longer configured" cleanup becomes opt-in:

- Removing a directory from `tome.toml` no longer triggers library deletion automatically.
- Library entries from the removed source enter an **unowned** state: `manifest.source_name: Option<DirectoryName> = None` (or sentinel).
- Two new commands manage the unowned lifecycle:
  - `tome adopt <skill> <directory>` — re-anchor an unowned skill to a configured directory (updates manifest source_name)
  - `tome forget <skill>` — explicitly delete an unowned skill from the library
- `tome status` and `tome doctor` surface the unowned set so the user can decide when to adopt or forget.
- Current cleanup behavior remains accessible via `tome sync --aggressive-cleanup` (see OQ-2).

## Open Question Resolutions

Each section: **Decision** (recommended), **Alternatives considered**, **Rationale**, **Risk**.

### OQ-1: Migration UX — auto, explicit command, or guarded prompt?

**Decision:** Auto-on-first-sync with an interactive confirmation banner that lists what will change.

**Alternatives:**
- A. Silent auto-migration on first v0.10 sync.
- B. Explicit `tome migrate-library` command, fail-loud if old shape detected by `sync`.
- C. Guarded prompt: detect old shape, show plan, require single y/n.

**Rationale:** The migration is mechanical and idempotent (replace symlinks with copies of their resolved content). Risk of surprise is real — first-time v0.10 users will see "converting 62 symlinks to copies" without warning. But B forces a step that 100% of users must run, which is friction without proportional safety. **C is the middle path:** detect on first sync, render summary table ("62 symlinks → real directories, ~30MB additional disk"), require y/n, persist consent in `machine.toml::migration_v010_acknowledged: true` so it doesn't prompt every sync.

**Risk:** A user with a partially-corrupted library (some symlinks broken from plugin uninstalls) gets surprised by which entries become missing during migration. Mitigation: **don't delete broken-symlink entries** during migration — leave them in `unowned` state for explicit `tome adopt`/`forget`.

### OQ-2: Edit-in-library semantics — managed skill edited locally

**Decision:** Detect on next sync via content_hash mismatch + `managed: true`. Prompt: **promote-to-fork (default)**, revert, or skip.

**Alternatives:**
- A. Refuse edits — `tome sync` overwrites managed library entries unconditionally. (Today's behavior, transitively.)
- B. Auto-revert without asking.
- C. Auto-fork without asking — promote to local, decoupled from upstream.
- D. Three-way merge.

**Rationale:** D is too complex for v0.10 (no good merge UX in CLI). A is hostile (silent data loss). B is hostile (silent data loss). C is friendlier but still surprising. **The interactive prompt mirrors the existing `tome fork` command's intent:** the user has already de-facto forked by editing; the prompt just makes it explicit. Default = "fork" (preserves the user's edit) since that's the safer outcome.

**Risk:** Triage friction in non-interactive `tome sync --no-input`. Resolution: in non-interactive mode, default to **skip with warning** ("managed skill X has local edits, not updating; run `tome fork X <dir>` or pass `--auto-fork-edits` to resolve"). Never silently overwrite user content.

### OQ-3: `tome sync` install policy — auto, prompt, or opt-in?

**Decision:** Prompt on first machine; remember consent in `machine.toml::auto_install_plugins: bool`; honor `--no-install` global override.

**Alternatives:**
- A. Always install missing plugins by default. New-machine onboarding is zero-touch.
- B. Always require `--install` flag. Surprising-action-prevention default.
- C. Prompt every sync. Annoying.
- D. Per-machine first-time prompt, then remember.

**Rationale:** A is dangerous on shared/CI machines where you don't want tome to side-effect-install plugins. B is friction for the library-as-dotfiles use case (you'd type `--install` every sync). **D balances:** new machine sees a one-time "Tome wants to install N missing plugins. Always install going forward? [Y/n/never]" prompt; future syncs honor the choice; can be reset by editing `machine.toml`.

**Risk:** A user accidentally answers "never" and then stops getting plugins. Mitigation: every sync surfaces "N plugins missing (auto-install disabled — re-enable in machine.toml or use `--install`)" so the state is always visible.

### OQ-4: Failure handling — install fails for some plugins

**Decision:** Partial-succeed with grouped failure summary (matching SAFE-01 / RemoveFailure pattern shipped in v0.8).

**Alternatives:**
- A. Bail entire sync on first install failure.
- B. Partial-succeed: install what we can, surface failures, continue distribute.
- C. Queue-and-retry with backoff.

**Rationale:** A is too brittle — one network blip leaves the user blocked. C overengineered for v0.10. **B matches the existing `tome remove` semantics shipped in v0.8 (SAFE-01):** aggregate `Vec<InstallFailure>`, exit non-zero, group by failure kind in the summary. The library copy serves as fallback for any plugin we couldn't install — distribution still works because the library content is intact.

**Risk:** User sees "8 of 10 plugins installed, 2 failed" and loses track. Mitigation: lockfile only updates entries that succeeded; failed entries stay at previous lockfile state; `tome doctor` surfaces them at every check until resolved.

### OQ-5: Per-skill version pinning location

**Decision:** Per-plugin pin in `tome.toml::[directories.claude-plugins.pins]`. **Defer the implementation** to a follow-up issue post-v0.10.

**Alternatives:**
- A. `[directories.claude-plugins.pins]` in `tome.toml` (portable across machines).
- B. `[directory.claude-plugins.pins]` in `machine.toml` (per-machine).
- C. Per-skill in lockfile (already exists; user manually edits).
- D. No pinning; lockfile-record-only.

**Rationale:** **D for v0.10.** The Claude CLI doesn't support `--version` flags; pinning is therefore advisory until upstream supports it. Lockfile records actual installed version; cross-machine drift surfaces as warning; user accepts "latest" as the v0.10 reproducibility level. **When (if) Claude Code adds version-pinned install,** A is the right home for cross-machine pinning (same plugin pinned to same version everywhere = portability) with B as override (Machine X needs older version due to local incompatibility).

**Risk:** A user expects pinning to work and is surprised it doesn't. Mitigation: documented constraint in `docs/src/architecture.md` and called out in the v0.10 changelog explicitly.

### OQ-6: Lockfile divergence across machines

**Decision:** Default = render the diff, prompt for confirm before applying. Opt-out via `--no-update`.

**Alternatives:**
- A. Auto-apply lockfile state on every machine (Cargo-shaped).
- B. Require explicit confirm before any plugin update.
- C. Auto-apply if `auto_install_plugins` is set (OQ-3).

**Rationale:** **B is too noisy** — every sync on a freshly-pulled dotfiles repo would prompt. **A is too aggressive** — silent updates can break user workflows. **The right shape:** if `auto_install_plugins: true` (set by OQ-3 consent), apply silently with progress output; if not, prompt with a per-skill diff summary. The diff summary itself is informative (you see which plugins changed, with versions).

**Risk:** Silent updates land plugin versions the user wasn't ready for. Mitigation: `tome sync` always exits with a "what changed" summary even on auto-install; lockfile diff is in `git log -p tome.lock` for auditability.

### OQ-7: CLI hardening bundling — fold review-followups into v0.10?

**Decision:** Yes, bundle. v0.10 = "Library-canonical + CLI hardening" as a coherent milestone.

**Alternatives:**
- A. v0.10 = library-canonical only; v0.10.x patch releases for review-followups.
- B. v0.10 = library-canonical + critical review-followups (the 4 architecture-critical ones); v0.10.1+ for the rest.
- C. v0.10 = library-canonical + ALL 19 review-followups + the older bug backlog (~30 issues).

**Rationale:** **C is right for the milestone shape we're after.** The library-canonical work touches `consolidate`, `manifest`, `lockfile`, `cleanup` — the same modules where review-followups #486 (`lib.rs::run` 500-line match), #487 (`config.rs` 3042 LOC split), #488 (`process::exit` bypass) live. Doing them together is more efficient than serial. The older bug backlog (#416 relocate cross-fs, #430 reassign drift, #433 manifest epoch-0, #447 browse Disable/Enable stubs, #454 wizard ANSI width, #456 configure_library default, #457 Config::save_checked tilde) is also strongly correlated — most touch the manifest / config / cleanup paths the v0.10 work refactors.

**Risk:** Milestone gets too large to ship cleanly. Mitigation: phase the milestone into intermediate cuts (alpha = library-canonical core, beta = + adapter, rc = + CLI hardening, final = + bug backlog). See ROADMAP draft below.

### OQ-8: GUI milestone naming — v0.11 or v1.0?

**Decision:** **GUI = v1.0** (kept). Library-canonical = v0.10.

**Alternatives:**
- A. GUI = v0.11; v1.0 deferred to future "we genuinely commit to API stability" cut.
- B. GUI = v1.0; library-canonical work makes v1.0 land on solid ground.

**Rationale:** **B.** The library-canonical work is exactly what v1.0 needs — stable Rust types feeding the GUI's IPC boundary, durable library that doesn't churn on upstream changes, predictable cross-machine reproducibility. Calling it v1.0 *because the GUI ships* is naming-ceremony; calling it v1.0 *because the underlying model is now stable* is honest. v0.10 lays the foundation, v1.0 ships the visible product (GUI) on top of it. The "Backward compat: None" disclaimer on `CLAUDE.md` will need to be revisited at v1.0 ratification — at that point, the public Rust types feeding `bindings.ts` become a real compatibility surface.

**Risk:** Locking in v1.0 framing creates expectations of stability that we're not fully ready for. Mitigation: at v1.0 ratification (post-v0.10), explicitly decide which surfaces are stable (the IPC types, the lockfile schema) vs which still allow breakage (CLI flags, library on-disk shape). Document in CLAUDE.md.

## Risks & Concerns

### Architecturally significant

1. **Update timing change for managed skills.** Today: plugin update → user's Claude Code immediately sees new content (via symlink). After v0.10: plugin update doesn't reach Claude Code skills until next `tome sync`. **This is a behavior change worth surfacing in the changelog.** Some users may rely on instant propagation; v0.10 makes them opt into upstream changes via sync.

2. **Disk usage growth.** ~5–50 MB depending on library size. Mostly markdown; not material on modern machines. Worth noting in the migration prompt.

3. **`tome backup` snapshots become more useful but bigger.** Today's backups capture broken symlinks for managed skills (useless on restore to a different machine). After v0.10 they capture full content. Backup integrity story actually improves; size grows proportionally.

### UX / discovery

4. **First-time-on-Machine-B confusion.** User pulls dotfiles, runs `tome sync`, sees "12 plugins missing — install? [Y/n]" prompt. Need very clear messaging that this is normal first-machine onboarding, not an error.

5. **Lockfile churn in git.** Every sync that detects drift updates `tome.lock`. Diffs become noisy if multiple machines update at different times. Mitigation: lockfile entries are deterministically ordered (BTreeMap); `git diff tome.lock` is readable; users learn to treat it like Cargo.lock.

### Implementation

6. **`claude` CLI being a hard dependency.** The adapter shells out to `claude plugin install`. If the user doesn't have `claude` on PATH, the adapter fails. Acceptable: tome already implicitly depends on Claude Code being installed (it reads `installed_plugins.json`); making it explicit is fine. Surface clearly: "claude not found on PATH — install Claude Code or remove `[directories.claude-plugins]` from tome.toml."

7. **Adapter test surface.** Shelling out to `claude plugin` makes unit testing hard. Mitigation: `MarketplaceAdapter` is a trait; production uses `ClaudeMarketplaceAdapter`; tests use `MockMarketplaceAdapter` with scripted `current_version`/`install` responses.
