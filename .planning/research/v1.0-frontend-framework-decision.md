# ADR: v1.0 Frontend Framework Decision

- **Status:** Accepted
- **Date:** 2026-05-27
- **Phase:** 25 (rust-core-extraction-tauri-integration-spike), Plan 25-06
- **Requirement:** D-GUI-04 (frontend framework choice — irreversible from Phase 26 onward)
- **Decision:** **React** (v19)

## Context

v1.0 wraps the stabilized library-canonical Rust types as Tauri commands and renders
them in a desktop GUI. The frontend framework choice is load-bearing: every UI phase
(26–31) builds on it, and a mid-milestone swap is not acceptable (STATE.md "Frontend
framework decision is load-bearing"; D-GUI-04). Per D-GUI-04 the shortlist was
**React / Solid / Svelte**, and the decision was deferred to a *built* spike so the
choice is made against the real `bindings.ts` and real `StatusReport` data rather than
assumptions.

To settle it, the same single-page `StatusReport` dashboard was built in all three
frameworks (`crates/tome-desktop/ui-react/`, `ui-solid/`, `ui-svelte/`), each consuming
the ONE canonical `crates/tome-desktop/ui/src/bindings.ts` via a Vite `resolve.alias`
(`@bindings` → `../ui/src/bindings.ts`) — no per-spike copy, preserving the single
source of truth (SC#3). Each spike rendered a live `StatusReport` from the dev machine's
real `tome_home` (resolved paths, library dir, configured directories with role/type
badges, skill counts, last-sync, health summary, unowned skills) by pattern-matching the
tauri-specta `Result<StatusReport, TomeError>` discriminated union. No list
virtualization and no interactions — that is Phase 26 (D-02).

## Scoring

Each candidate scored 1–5 across the four D-03 criteria:

1. **bindings.ts ergonomics** — how naturally the generated typed commands and the
   `Result<StatusReport, TomeError>` discriminated union narrow in the framework's idioms.
2. **bundle size + cold-start TTI** — production `vite build` gzip size and webview
   cold-start.
3. **dev-loop speed** — HMR latency, error quality, type-check speed.
4. **ecosystem fit for v1.0 reqs** — off-the-shelf maturity for the upcoming
   virtualized 2000-row list (VIEW-02 / NF-01 @ 60fps), keyboard + VoiceOver
   accessibility (NF-02), and macOS-HIG-aligned components (NF-03).

| Criterion (1-5)                                          | React | Solid | Svelte |
|----------------------------------------------------------|:-----:|:-----:|:------:|
| 1. bindings.ts ergonomics                                |   5   |   3   |   4    |
| 2. bundle size + cold-start                              |   2   |   5   |   4    |
| 3. dev-loop speed                                        |   4   |   4   |   4    |
| 4. ecosystem fit (NF-01 virtualization / NF-02 a11y / NF-03 HIG) |   5   |   3   |   4    |
| **TOTAL**                                                | **16**| **15**| **16** |

### Measured data

- **Production bundle (gzip, identical dashboard):**
  - Solid: **6.20 kB**
  - Svelte: **15.85 kB**
  - React: **62.29 kB**
- **Type-check (0 errors all three):**
  - React: `tsc` 0.75s
  - Solid: `tsc` 0.56s
  - Svelte: `svelte-check` 1.04s

### Per-criterion notes

- **bindings.ts ergonomics (1):** tauri-specta emits a `Result<StatusReport, TomeError>`
  tagged union (`{ status: "ok", data } | { status: "error", error }`). React narrows it
  with a plain `if (res.status === "ok")` inside `useEffect` + `useState`, and the
  generated types flow through with zero casts — the most natural fit (5). Svelte's
  reactive `$state`/`await` blocks handled it cleanly but the narrowing read slightly
  less directly inside `{#await}`/`{#if}` markup (4). Solid's `<Show>` / `createResource`
  control flow fought the union narrowing — the discriminant had to be re-checked inside
  the rendered branch because `<Show when={...}>` does not narrow the outer binding (3).
- **bundle size + cold-start (2):** Solid wins decisively (6.20 kB, fine-grained
  reactivity, no VDOM) (5); Svelte close behind (15.85 kB, compiles away its runtime) (4);
  React's runtime + react-dom dominates at 62.29 kB — ~10× Solid, ~4× Svelte (2). For the
  static `StatusReport` dashboard cold-start was imperceptible in all three; the gap will
  matter more under the NF-01 2000-row budget, which is why this is scored honestly low
  for React.
- **dev-loop speed (3):** All three gave sub-second HMR and clear errors against the
  Vite 6 dev server; type-check times are within ~0.5s of each other. No meaningful
  differentiator — tie at 4.
- **ecosystem fit (4):** This is the compounding criterion across Phases 26–31. React has
  the most mature off-the-shelf support for the load-bearing upcoming reqs: TanStack
  Virtual for NF-01 2000-row virtualization @ 60fps, React Aria / Radix for NF-02
  keyboard + VoiceOver accessibility, and broad macOS-HIG-aligned component coverage
  (NF-03) — lowest-risk for an irreversible choice (5). Svelte's virtualization and a11y
  libraries are maturing but thinner (4). Solid has the smallest ecosystem and the
  thinnest off-the-shelf a11y/HIG/virtualization coverage (3).

## Decision

**React** is chosen for v1.0.

React and Svelte tied at 16 points total, but **React wins the two criteria that compound
across Phases 26–31** — bindings.ts ergonomics (1) and ecosystem fit (4):

- The tauri-specta `Result<StatusReport, TomeError>` union narrows most naturally in React
  (`if (res.status === "ok")`), and this pattern recurs in every command call across all
  remaining phases.
- The load-bearing upcoming requirements have the most mature, lowest-risk React ecosystem
  support: NF-01 2000-row virtualization @ 60fps (TanStack Virtual), NF-02 keyboard +
  VoiceOver accessibility (React Aria / Radix), NF-03 macOS HIG components.

For an **irreversible** decision, minimizing ecosystem risk on the two compounding axes
outweighs React's larger bundle. The bundle gap is real but bounded — the GUI is a local
desktop app, not a latency-sensitive web page — and NF-01's 60fps budget is met at the
list-virtualization layer (render only visible rows) rather than by raw framework
reactivity, which neutralizes much of Solid's raw-engine advantage.

### Honest counter-arguments (recorded for the record)

- **Svelte** tied at 16 with a ~4× smaller bundle (15.85 kB vs 62.29 kB) and the cleanest
  authoring experience. It loses only on a11y/virtualization ecosystem maturity — the very
  axis that is hardest to recover from after the irreversible cut. If React's ecosystem
  edge proves illusory (see invalidation conditions), Svelte is the strongest fallback.
- **Solid** had the smallest bundle (6.20 kB) and the best raw 60fps engine, but scored
  weakest on the two compounding criteria: union narrowing fought its `<Show>` control
  flow, and its off-the-shelf a11y / HIG / virtualization coverage is the thinnest of the
  three. Best raw performance, highest ecosystem risk for an irreversible v1.0 commit.

## What would invalidate this choice

This decision is irreversible from Phase 26 onward by policy, but the following would
constitute evidence that the premise was wrong (to be captured as a milestone-level
re-evaluation, NOT a mid-milestone swap):

1. **NF-01 fails in React.** If the 2000-row virtualized list cannot hold 60fps in the
   webview using React + TanStack Virtual even after standard optimization
   (memoization, windowing, avoiding unnecessary re-renders), React's bundle/runtime
   overhead would have been the wrong trade and Solid/Svelte's raw-engine advantage would
   have mattered.
2. **React's a11y ecosystem advantage doesn't materialize.** If React Aria / Radix do not
   in practice deliver NF-02 keyboard + VoiceOver accessibility more cheaply than Svelte's
   options, the primary rationale (criterion 4) collapses and the tie with Svelte should
   have been broken on bundle size instead.
3. **bindings.ts ergonomics regress.** If a future tauri-specta version changes the
   `Result` union shape such that React's narrowing advantage disappears (e.g. emitting a
   throwing API instead of a tagged union), criterion 1's edge is neutralized.
4. **Bundle/cold-start becomes user-visible.** If the shipped app's cold-start or memory
   footprint (DIST, Phase 31) measurably degrades UX in a way Solid/Svelte would have
   avoided, the bundle trade-off was mispriced.

If none of these trigger through the alpha (Phase 26) and beta (Phase 28) cuts, the
choice is considered validated.

## Consequences

- The winning React spike is collapsed into `crates/tome-desktop/ui/` (co-located with
  the canonical `src/bindings.ts`); the `@bindings` alias is repointed to the now-sibling
  `./bindings`. The losing `ui-solid/` and `ui-svelte/` spikes are deleted.
- `tauri.conf.json` points permanently at `ui/` (frontendDist / devUrl /
  beforeDevCommand / beforeBuildCommand).
- The single canonical `bindings.ts` stays at `crates/tome-desktop/ui/src/bindings.ts`;
  the CI gen-bindings freshness gate path is unchanged.
- D-GUI-04 is updated to name React in `.planning/REQUIREMENTS.md` (canonical) and
  `.planning/milestones/v1.0-REQUIREMENTS.md` (archive copy).
