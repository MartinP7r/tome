03 Decision Log

Key design decisions from the past 2 weeks — the reasoning that evaporates first.

SkillName newtype with lenient-now, strict-later validation
Rejects only empty names and path separators today, but warns on non-[a-z0-9-]+ names. This avoids breaking existing skills while signaling the v0.3 hard requirement. The warning gives users migration time.
Considered: strict validation from day one (would break existing skills); no validation (would let path traversal bugs slip through)
TargetMethod enum encodes required paths
Symlink { skills_dir } and Mcp { mcp_config } make invalid states unrepresentable — a symlink target must have a skills_dir. Previously, both fields were optional on a flat struct with a method string, allowing silent misconfiguration.
Considered: keeping flat struct with runtime validation (more error-prone); trait-based polymorphism (premature for 2 variants)
BTreeSet for exclusion list instead of Vec<String>
Deterministic serialization order in TOML output. O(log n) contains() check during discovery loop. Free deduplication. Uses the SkillName type for validation at parse time.
Considered: HashSet (non-deterministic order breaks TOML diff readability); Vec with manual dedup (more code)
Skip distribution when skill originates from target directory
When ~/.claude/skills is both a source and a target, the library has a symlink pointing into the target dir. Creating another symlink in the target pointing back to the library would be circular. Now detected via canonicalize + starts_with.
Considered: blocking this at config time (too restrictive — the overlap is often intentional); ignoring it (silent circular symlinks)
MCP server caches skills at startup
Discovery is a filesystem walk — doing it per-request would be slow and noisy. The trade-off is that skills added after server start aren't visible until restart. Acceptable for a v0.1 where the server is invoked per-session.
Considered: lazy discovery per request (too slow); fsnotify watcher (deferred to v0.6)
Git commit prompt after sync (interactive only)
If the library dir is a git repo with uncommitted changes post-sync, offer to git add . && git commit with a summary message like "tome sync: 3 created, 1 updated". Skipped in dry-run, quiet, and non-TTY contexts to prevent automation breakage.
Considered: always auto-committing (surprising behavior); never offering (users forget to commit library changes)

05 Mental Model Essentials

The 10 things you need to hold in your head to work on this project effectively.

01
Sync pipeline is the spine. Every sync (whether from tome sync or tome init) follows discover → consolidate → distribute → cleanup. Understanding this flow is understanding the whole project.
02
Everything is symlinks. Library entries are symlinks to original skill directories. Target entries are symlinks to library entries. Originals are never moved or copied. This is the core invariant.
03
Two source types, two distribution methods. Sources: ClaudePlugins (reads installed_plugins.json v1/v2) and Directory (walkdir scan for */SKILL.md). Distribution: Symlink (directory links) and Mcp (JSON config entries).
04
First source wins. When two sources have a skill with the same name, the one listed first in config takes priority. This is deterministic and intentional — source ordering is the conflict resolution strategy.
05
dry_run threads through everything. Every operation accepts dry_run: bool and counts what would change without touching the filesystem. Result structs report the same counts either way.
06
Targets struct is hardcoded. config::Targets has four named Option<TargetConfig> fields. Adding a new target means modifying the struct, iter(), iter_mut(), wizard, and serde layer. v0.2 replaces this with Vec<Target>.
07
Config lives at ~/.config/tome/config.toml. Tilde expansion happens at load time in expand_tildes(). Config::load_or_default gracefully handles missing files (first-run scenario).
08
Warnings over errors. Missing source paths, broken symlinks, and non-conventional skill names produce eprintln! warnings, not hard errors. The philosophy: partial success is better than no sync.
09
Tests use tempfile::TempDir. Every test creates isolated temp directories. No cleanup needed — TempDir drops on scope exit. Integration tests exercise the binary via assert_cmd.
10
macOS /var symlink pitfall. On macOS, /var → /private/var. Both cleanup.rs and doctor.rs canonicalize library_dir and check starts_with against both the original and canonical path to handle this.

06 Cognitive Debt Hotspots

Areas where understanding is weakest or technical debt is accumulating.

High
Hardcoded Targets struct blocks new tool support
Every new AI tool requires touching Targets struct fields, iter(), iter_mut(), serde layer, wizard match arms, and integration tests. With the AI tool landscape growing fast, this is the biggest scaling bottleneck.
Suggestion: Prioritize the v0.2 connector architecture. Even a partial migration (Vec<Target> without the full trait) would unblock adding tools without code changes.
Medium
wizard.rs is the longest module at 597 lines
Contains step configuration, known source scanning, overlap detection, exclusion picker, and summary display. Each wizard step is a standalone function, but the module has no internal organization beyond function ordering.
Suggestion: Extract KNOWN_SOURCES and find_known_sources_in() into a shared defaults.rs module. The overlap detection logic could move to config.rs as a validation method.
Medium
offer_git_commit has no TTY integration test coverage
The new offer_git_commit() function in lib.rs:256 can only be tested in non-TTY contexts (which skip the prompt). The actual interactive path — where dialoguer::Confirm prompts the user — is untested. CI also runs without a TTY.
Suggestion: Consider adding a --yes / --auto-commit flag that bypasses the interactive prompt, making the commit path testable in CI.
Low
MCP server has no way to refresh skill cache
TomeServer::new() in mcp.rs:19 calls discover_all() once at startup. Skills added after the server starts are invisible until restart. Acceptable for now but will bite when tome is used as a long-running MCP server.
Suggestion: Add an internal refresh_skills tool to the MCP server that re-runs discovery. No protocol change needed — just a new tool endpoint.
Low
Serde layer for TargetConfig uses a shadow RawTargetConfig
config.rs:131–210 maintains a parallel RawTargetConfig struct with manual Serialize/Deserialize impls that convert to/from the real TargetConfig. This works but doubles the surface area for target config changes.
Suggestion: When migrating to Vec<Target> in v0.2, consider using #[serde(tag = "method")] on an enum to eliminate the shadow struct entirely.
Low
Stale branch fix/quick-wins-2 lingering
Unmerged branch from an earlier sprint. May contain partially-applied fixes or overlap with work that landed via other PRs.
Suggestion: Diff against main and either cherry-pick remaining useful changes or delete the branch.
