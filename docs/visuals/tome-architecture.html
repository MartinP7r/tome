tome — Architecture

A CLI tool that discovers AI coding skills (SKILL.md files) from multiple sources and distributes them to coding assistants like Claude Code, Antigravity, Codex, and OpenClaw. One library, many targets — originals are never moved or copied, only symlinked.

Crate
Rust workspace with two binaries: tome (CLI) and tome-mcp (MCP server)
Philosophy
Symlinks everywhere — originals stay in place, library and targets reference them
Sources
ClaudePlugins (installed_plugins.json) and Directory (walkdir scan for SKILL.md)
Targets
Claude Code, Antigravity, Codex, OpenClaw — via symlinks or MCP config injection
Sync Pipeline

The core flow. Both tome sync and tome init invoke sync() in lib.rs. Four stages execute in sequence — each is idempotent and supports dry_run mode.

Sources → Discover → Consolidate → Library → Distribute → Targets → Cleanup
1. Discover
Scans configured sources for */SKILL.md directories. First source wins on name conflicts. Exclusion list filters unwanted skills.
2. Consolidate
Creates symlinks from library to each discovered skill's original location. Unchanged links are skipped; stale links are updated.
3. Distribute
Pushes library skills to each enabled target. Symlink targets get directory symlinks; MCP targets get a tome server entry.
4. Cleanup
Removes broken symlinks from library (source deleted) and stale symlinks from targets (skill removed from library).

Sources

Sources tell tome where to find skills. Each source is scanned during the Discover stage. The first source to claim a skill name wins — later duplicates are silently skipped.

ClaudePlugins
Claude Plugin Cache
Reads installed_plugins.json (v1 or v2 format) from Claude's plugin cache directory. Each cached plugin is checked for a SKILL.md file at its root.
  • Default path: ~/.claude/plugins/cache
  • Scans: JSON manifest → iterate cached plugins
  • Handles: Both v1 (flat list) and v2 (structured) formats
  • Skill name: Derived from plugin directory name
Directory
Flat Directory Scan
Walks a directory looking for */SKILL.md — any subdirectory containing a SKILL.md file is treated as a skill. Simple and universal.
  • Common paths: ~/.claude/skills, ~/.codex/skills, ~/.gemini/antigravity/skills
  • Scans: One level deep — dir/*/SKILL.md
  • Handles: Any directory structure with SKILL.md children
  • Skill name: Subdirectory name (validated as SkillName)
Auto-discovery in wizard
During tome init, the wizard probes known locations and offers to add them: ~/.claude/plugins/cache (ClaudePlugins), ~/.claude/skills, ~/.codex/skills, ~/.gemini/antigravity/skills (all Directory type). Users can also add custom directories interactively.
Distribution

Two methods for pushing skills to targets. The method is configured per-target in the TOML config. Each method is a variant of the TargetMethod enum — invalid combinations are impossible at the type level.

Symlink
Directory Symlinks
Creates a symlink in the target's skills directory pointing to each skill in the library. The target tool discovers skills by scanning its skills directory as usual.
  • Chain: target/skill → library/skill → original
  • Used by: Claude Code, Antigravity, Codex, OpenClaw
  • Config key: skills_dir
  • Cleanup: Removes symlinks pointing to deleted library entries
MCP
MCP Config Injection
Writes a "tome" server entry into the target's .mcp.json file. The target tool connects to tome's MCP server, which exposes skills via list_skills and read_skill tools.
  • Protocol: MCP over stdio (rmcp crate)
  • Used by: Claude Code (as alternative to symlinks)
  • Config key: mcp_config (path to .mcp.json)
  • Cleanup: Removing the MCP entry is not yet automated
Two-hop symlink chain
With the Symlink method, each skill has a two-hop symlink chain. The original file is never moved.

~/.claude/skills/my-skill → ~/.tome/skills/my-skill → /original/path/my-skill

The library is the single source of truth for which skills are active. Cleanup walks each hop and removes links whose target no longer exists.
Filesystem Layout

Three layers: config tells tome what to do, the library collects active skills, and targets receive them.

Config
~/.tome/tome.toml
Defines sources (what to scan), targets (where to push), exclude list, and library path. Config::load_or_default handles missing files gracefully.
Library
~/.tome/skills/
Central registry. One symlink per active skill, pointing to the original source. The library is the single source of truth.
Targets
Per-tool skills directories.
Each enabled target receives symlinks from library entries, or an MCP config entry pointing to the tome server.
Key Types

Core domain types that make invalid states unrepresentable. The type system enforces naming rules, source-target relationships, and distribution method constraints.

TargetMethod — Distribution enum
pub enum TargetMethod {
    Symlink {
        skills_dir: PathBuf,
    },
    Mcp {
        mcp_config: PathBuf,
    },
}

// Makes invalid method + config combos
// impossible at the type level
// skills_dir only with Symlink
// mcp_config only with Mcp
TargetConfig — Per-target config
pub struct TargetConfig {
    pub enabled: bool,
    pub method: TargetMethod,
}

impl TargetConfig {
    pub fn skills_dir(&self) -> Option<&Path> {
        match &self.method {
            TargetMethod::Symlink { skills_dir }
                => Some(skills_dir),
            _ => None,
        }
    }
}
Config — Top-level configuration
pub struct Config {
    pub sources: Vec<Source>,       // what to scan
    pub targets: Targets,           // where to push (hardcoded struct, not Vec)
    pub library_dir: PathBuf,       // central library path
    pub exclude: BTreeSet<SkillName>, // skills to skip
}

pub struct Source {
    pub name: String,
    pub path: PathBuf,
    pub source_type: SourceType,    // ClaudePlugins or Directory
}

pub struct Targets {
    pub antigravity: Option<TargetConfig>,
    pub claude: Option<TargetConfig>,
    pub codex: Option<TargetConfig>,
    pub openclaw: Option<TargetConfig>,
}

Design note
The Targets struct uses named fields rather than a generic Vec<Target>. This keeps the current implementation simple — each target has known defaults and wizard behavior. The v0.2 roadmap plans to replace this with a connector trait and Vec<Target> for extensibility.
